# AGENTS.md

이 문서는 이 저장소에서 작업할 때 항상 우선 참고하는 로컬 작업 규칙이다.
애매할 때는 "짧고 실행 가능하며 검증 가능한 규칙"으로 해석한다.

## 1. 최우선 규칙

1. 모든 응답은 한국어로 작성한다.
2. 내가 요구하는 걸 완벽하게 이해할 때까지 질문해. 그리고 100% 이해했을 때 코드짜기를 시작한다.
3. 코드 주석은 한국어로 작성한다.
4. 사용자가 "질문"을 한 경우에는 바로 수정하지 말고 먼저 답변한다.
5. 질문에 답하는 과정에서 수정 필요 사항을 발견하면, 답변 후 문제점과 수정 필요 여부를 함께 알린다.
6. 사용자가 명시적으로 수정이나 구현을 요청한 경우에는 별도 계획 문서보다 실제 코드 수정과 검증을 우선한다.

## 2. 작업 전 확인

1. 작업 시작 전 `docs/` 폴더의 관련 문서를 먼저 읽고 구조와 도메인 로직을 파악한다.
2. `znotes/` 폴더의 파일은 사용자 메모로 간주한다.
3. `znotes/`는 사용자가 명시적으로 지시한 경우에만 읽거나 참조한다.
4. 리팩토링이나 구조 변경 시 `Dockerfile`, 배포 스크립트, 실행 엔트리 포인트 영향 여부를 반드시 확인한다.

## 3. 구현 원칙

1. 암묵적인 fallback, silent default, 임의 보정 로직을 만들지 않는다.
2. 필수 설정값이 없으면 즉시 명확한 에러를 발생시킨다.
3. 함수는 가능하면 짧고 한 가지 역할만 하게 유지한다.
4. 하나의 함수가 여러 책임을 가지면 하위 함수로 분리하는 쪽을 우선 검토한다.
5. 문법 호환성 문제를 이유로 코드 수준을 낮추지 않는다.
6. 예를 들어 실행 환경 문제를 맞추기 위해 최신 타입 힌트를 구문상 구버전 스타일로 되돌리는 식의 수정은 하지 않는다.
7. 대신 올바른 실행 환경과 프로젝트가 의도한 Python 버전을 찾는다.

## 4. 실행 및 검증

1. Python 실행 시 시스템 `python`, `python3`를 당연하게 가정하지 않는다.
2. 먼저 프로젝트의 가상환경 경로를 확인하고 그 Python을 우선 사용한다.
3. 코드 수정 후에는 `.pre-commit-config.yaml`을 확인하고, 정의된 린트/포맷팅/검사 규칙 기준으로 검증한다.
4. 검증 명령은 가능한 한 수정 범위와 직접 관련된 것부터 최소 단위로 실행한다.

## 5. 작업 방식

1. 기본 원칙은 순차 처리다.
2. 불필요한 병렬 처리는 피한다.
3. 단, 다수의 백테스트를 수행하는 튜닝 작업처럼 병렬화가 본질적인 경우는 예외로 한다.
4. Implementation Plan, Task, Walkthrough 같은 중간 산출물 문서는 기본적으로 만들지 않는다.
5. 다만 사용자 요청 또는 작업상 꼭 필요한 경우에만 짧게 작성한다.

## 6. Git 규칙

1. 사용자가 명시적으로 요청하지 않으면 `git commit`과 `git push`는 실행하지 않는다.
2. 코드 변경 작업을 마치면 바로 사용할 수 있는 짧은 커밋 메시지 스타일 요약을 함께 제공한다.
3. 커밋 요약은 한국어로, 가능하면 20단어 이내로 작성한다.

## 7. 해석 우선순위

1. 사용자의 현재 요청
2. 이 `AGENTS.md`
3. 관련 `docs/` 문서
4. 그 외 코드와 기존 구현 관례

## 8. 이 문서의 의도

1. 쓸데없이 긴 설명보다 바로 수정 가능한 판단 기준을 제공한다.
2. 모호한 규칙보다 실제 작업 중 충돌이 적은 규칙을 남긴다.
3. 새 규칙을 추가할 때는 "언제", "무엇을", "예외는 무엇인지"가 바로 드러나게 쓴다.
